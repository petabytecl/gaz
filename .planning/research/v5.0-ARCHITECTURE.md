### Component Responsibilities

| Component | Responsibility | Status in v5.0 |
|-----------|----------------|----------------|
| **Vanguard Transcoder** | Protocol detection, transcoding (REST↔gRPC↔Connect↔gRPC-Web), single `http.Handler` entry point | NEW — replaces `gateway/` |
| **gRPC Server** | Native gRPC service hosting, registrar pattern, interceptor chain | MODIFIED — wrapped by `vanguardgrpc.NewTranscoder`, no longer owns its own listener |
| **Connect Registrar** | Register Connect service handlers, return `(path, http.Handler)` pairs | NEW — mirrors `grpc.Registrar` |
| **Connect Interceptors** | Per-handler interceptor chain for Connect services | NEW — separate from gRPC interceptors |
| **UnknownHandler** | Non-RPC HTTP routes (health endpoints, metrics, static assets) | NEW — registered via `vanguard.WithUnknownHandler` |
| **CORS Middleware** | Cross-origin request handling | MIGRATED — moves from gateway to wrap Vanguard handler |
| **HTTP Server** | Generic HTTP server with timeouts, serves the composed handler | MODIFIED — serves Vanguard handler instead of gateway handler |
| **Standalone HTTP** | HTTP-only server for non-RPC apps | UNCHANGED — kept for apps that don't need gRPC/Connect |
| **OTEL Module** | OpenTelemetry TracerProvider | UNCHANGED |

### Structure Rationale

- **`vanguard/`**: Owns the single-port HTTP server, Vanguard transcoder composition, and h2c setup. This is the new "glue" package that replaces `gateway/`.
- **`grpc/`**: Still owns `*grpc.Server` creation and service registration but **no longer owns a listener**. The gRPC server is wrapped by `vanguardgrpc.NewTranscoder` and served through Vanguard.
- **`connect/`**: New package mirroring `grpc/` patterns. Defines `ConnectRegistrar`, collects handlers, applies Connect-specific interceptors.
- **`gateway/`**: Deprecated. Kept during migration phase for backward compatibility but marked for removal.
- **`http/`**: Unchanged. Serves apps that only need a plain HTTP server (no gRPC/Connect).

## Architectural Patterns

### Pattern 1: Dual Registrar (gRPC + Connect)

**What:** Two parallel registrar interfaces for service registration, auto-discovered via `di.ResolveAll[T]`.
**When to use:** Always — this is the core extension point for v5.0.
**Trade-offs:** Two interfaces to implement vs. one, but each is protocol-appropriate and type-safe.

**Example:**
```go
// Existing gRPC registrar (unchanged)
type Registrar interface {
    RegisterService(grpc.ServiceRegistrar)
}

// New Connect registrar
type ConnectRegistrar interface {
    // Returns the path prefix and handler for this service.
    RegisterConnectService(interceptors []connect.Interceptor) (string, http.Handler)
}

// Implementation example
type GreeterService struct{}

// Implements grpc.Registrar — native gRPC
func (s *GreeterService) RegisterService(srv grpc.ServiceRegistrar) {
    greetv1.RegisterGreeterServiceServer(srv, s)
}

// Implements connect.ConnectRegistrar — Connect protocol
func (s *GreeterService) RegisterConnectService(interceptors []connect.Interceptor) (string, http.Handler) {
    return greetv1connect.NewGreeterServiceHandler(s, connect.WithInterceptors(interceptors...))
}
```

### Pattern 2: Vanguard as Composition Root

**What:** Vanguard Transcoder is the single `http.Handler` that composes all services. The `vanguard/` module builds it from discovered gRPC and Connect registrars.
**When to use:** Always in unified server mode.
**Trade-offs:** Single point of failure for all protocols, but simplifies deployment and networking. Vanguard is well-tested by Buf.

**Example:**
```go
func NewTranscoder(
    grpcServer *grpc.Server,
    connectRegistrars []ConnectRegistrar,
    connectInterceptors []connect.Interceptor,
    unknownHandler http.Handler,
) (*vanguard.Transcoder, error) {
    var services []*vanguard.Service

    // 1. Wrap gRPC server — auto-creates services for each registered gRPC service
    grpcTranscoder := vanguardgrpc.NewTranscoder(grpcServer)
    services = append(services, grpcTranscoder.Services()...)

    // 2. Wrap Connect handlers
    for _, reg := range connectRegistrars {
        path, handler := reg.RegisterConnectService(connectInterceptors)
        svc := vanguard.NewService(path, handler)
        services = append(services, svc)
    }

    // 3. Build transcoder with unknown handler for non-RPC routes
    opts := []vanguard.TranscoderOption{}
    if unknownHandler != nil {
        opts = append(opts, vanguard.WithUnknownHandler(unknownHandler))
    }

    return vanguard.NewTranscoder(services, opts...)
}
```

### Pattern 3: h2c for Single-Port gRPC

**What:** HTTP/2 cleartext (h2c) enables gRPC over HTTP/2 without TLS on the same port as HTTP/1.1 traffic.
**When to use:** Always for the unified port in development/internal services. Production behind a TLS-terminating proxy.
**Trade-offs:** Requires Go 1.25+ `http.Protocols` API (native h2c support). Earlier Go versions need `golang.org/x/net/http2/h2c` wrapper.

**Example:**
```go
// Go 1.25+ (gaz requires Go 1.26+, so this is our path)
func newHTTPServer(handler http.Handler, cfg *Config) *http.Server {
    srv := &http.Server{
        Addr:    fmt.Sprintf(":%d", cfg.Port),
        Handler: handler,
    }

    // Enable h2c (HTTP/2 without TLS)
    var protos http.Protocols
    protos.SetHTTP1(true)
    protos.SetHTTP2(true)
    protos.SetUnencryptedHTTP2(true) // This is h2c
    srv.Protocols = &protos

    return srv
}
```

### Pattern 4: Middleware at HTTP Layer

**What:** Cross-cutting concerns (CORS, OTEL, logging) applied as `http.Handler` middleware wrapping the Vanguard transcoder, rather than at the gRPC interceptor or Connect interceptor level.
**When to use:** For concerns that apply to ALL protocols uniformly (CORS, request ID, top-level logging).
**Trade-offs:** Protocol-specific middleware (auth token extraction, gRPC status codes) still needs interceptor-level handling.

**Example:**
```go
func buildHandler(transcoder http.Handler, cfg *Config) http.Handler {
    var handler http.Handler = transcoder

    // CORS wraps everything
    if cfg.CORS.Enabled {
        handler = cors.New(cors.Options{
            AllowedOrigins:   cfg.CORS.AllowedOrigins,
            AllowedMethods:   cfg.CORS.AllowedMethods,
            AllowedHeaders:   cfg.CORS.AllowedHeaders,
            AllowCredentials: cfg.CORS.AllowCredentials,
        }).Handler(handler)
    }

    return handler
}
```

### Key Data Flows

1. **Service Registration (startup):** `gaz.App.Start()` → DI resolves `grpc.Registrar[]` and `connect.ConnectRegistrar[]` → gRPC registrars register on `*grpc.Server` → Connect registrars produce `(path, handler)` pairs → Vanguard transcoder composes all services → HTTP server starts with composed handler.

2. **Protocol Detection (per-request):** HTTP request arrives → Vanguard inspects `Content-Type` header and path → `application/grpc` routes to gRPC server → `application/proto` or `application/json` on Connect paths routes to Connect handler → REST paths matching `google.api.http` annotations transcoded to gRPC → Unknown paths fall through to `UnknownHandler`.

3. **Shutdown:** `gaz.App.Stop()` → HTTP server graceful shutdown (stops accepting new connections) → In-flight requests drain → gRPC server `GracefulStop()` → Connect handlers complete naturally (they're just HTTP handlers).

## Scaling Considerations

| Scale | Architecture Adjustments |
|-------|--------------------------|
| Dev / 0-100 users | Single port, h2c, no TLS. Vanguard handles everything. |
| 100-10K users | TLS termination at load balancer. Vanguard still single binary. |
| 10K+ users | Consider separating gRPC (internal) and REST/Connect (public) behind different ingress rules. Same binary, different config. |

### Scaling Priorities

1. **First bottleneck — Connection handling:** h2c multiplexing helps. HTTP/2 reuses connections. Monitor with Vanguard's built-in metrics.
2. **Second bottleneck — Transcoding overhead:** REST→gRPC transcoding adds CPU. Monitor per-protocol latency. If REST transcoding is hot, consider Connect clients directly (no transcoding needed).

## Anti-Patterns

### Anti-Pattern 1: Dual Listeners for Same Protocol

**What people do:** Keep a separate gRPC listener alongside Vanguard "for backward compatibility."
**Why it's wrong:** Two listeners for gRPC creates confusion about which port to use, doubles health check surface, and splits interceptor chains.
**Do this instead:** Route ALL gRPC through Vanguard. Use `vanguardgrpc.NewTranscoder` to wrap the `*grpc.Server`.

### Anti-Pattern 2: Mixing Interceptor Layers

**What people do:** Apply authentication in gRPC interceptors AND Connect interceptors AND HTTP middleware.
**Why it's wrong:** Triple execution, inconsistent behavior, hard to debug.
**Do this instead:** Choose ONE layer per concern. Auth at interceptor level (protocol-specific token extraction). CORS at HTTP middleware level (protocol-agnostic). Logging at HTTP middleware for access logs, interceptor for RPC-specific logs.

### Anti-Pattern 3: Codegen Gateway Alongside Vanguard

**What people do:** Keep grpc-gateway codegen AND Vanguard, using both for REST.
**Why it's wrong:** Duplicate REST handling. grpc-gateway's codegen is replaced by Vanguard's annotation-based transcoding.
**Do this instead:** Remove grpc-gateway entirely. Use `google.api.http` annotations in proto files — Vanguard reads them directly from the service descriptors.

### Anti-Pattern 4: Global Connect Interceptors When Per-Service Needed

**What people do:** Apply ALL Connect interceptors to ALL Connect services.
**Why it's wrong:** Some services (health, reflection) shouldn't have auth interceptors. Connect doesn't have a global interceptor mechanism — interceptors are per-handler.
**Do this instead:** Let `ConnectRegistrar` implementations choose which interceptors to apply via the `interceptors []connect.Interceptor` parameter. Provide interceptor groups (e.g., "public", "authenticated") in DI.

## Integration Points

### External Services

| Service | Integration Pattern | Notes |
|---------|---------------------|-------|
| Buf Schema Registry | Proto descriptors at build time | Vanguard reads `google.api.http` annotations from compiled descriptors |
| OpenTelemetry | HTTP middleware + interceptors | OTEL HTTP middleware for overall tracing; gRPC/Connect interceptors for RPC-level spans |
| Health Checks | UnknownHandler routes | `/healthz`, `/readyz` served via Vanguard's unknown handler |
| gRPC Reflection | gRPC server option | Enable on `*grpc.Server` before wrapping with Vanguard — works through Vanguard automatically |

### Internal Boundaries

| Boundary | Communication | Notes |
|----------|---------------|-------|
| `vanguard/` ↔ `grpc/` | `vanguardgrpc.NewTranscoder(grpcServer)` | Vanguard wraps the completed gRPC server. gRPC module must finish registering services BEFORE Vanguard builds. |
| `vanguard/` ↔ `connect/` | `vanguard.NewService(path, handler)` per registrar | Each Connect registrar produces a service. Vanguard collects them all. |
| `vanguard/` ↔ `http/` | `http.Handler` composition | Vanguard transcoder is the handler; HTTP server owns the listener. |
| `grpc/` ↔ `connect/` | **No direct communication** | Services may implement both interfaces, but the packages don't depend on each other. |
| `server/module.go` ↔ all sub-modules | DI module composition | Top-level module ensures correct startup order via `gaz.For[T]().Eager()` and dependency ordering. |

## Migration Strategy (gateway/ → vanguard/)

### Phase Approach

1. **Phase A — Add vanguard/ and connect/ packages.** Gateway still exists and works. Users can opt-in to Vanguard.
2. **Phase B — Default to Vanguard.** `server.Module()` uses Vanguard by default. Gateway available via `server.LegacyModule()`.
3. **Phase C — Remove gateway/.** Breaking change. Document migration guide.

### Compatibility Considerations

- **gRPC Registrar interface is unchanged.** Services implementing `grpc.Registrar` work identically.
- **Gateway Registrar has no equivalent.** Services using `gateway.Registrar` must migrate to proto annotations for REST or implement `connect.ConnectRegistrar` for Connect protocol.
- **Port configuration changes.** Gateway config (`gateway.port`, `gateway.grpc_target`) replaced by Vanguard config (`vanguard.port`). gRPC config drops `port` field.
