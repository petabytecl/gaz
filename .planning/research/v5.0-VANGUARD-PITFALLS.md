# Pitfalls Research: Vanguard + Connect-Go Integration

**Domain:** gRPC-to-Vanguard/Connect migration in Go DI framework
**Researched:** Fri Mar 06 2026
**Confidence:** HIGH (verified against official docs, Context7, GitHub issues, and existing gaz codebase)

## Critical Pitfalls

### Pitfall 1: Interceptor Incompatibility — gRPC vs Connect Are Different Worlds

**What goes wrong:**
Existing gRPC interceptors (`grpc.UnaryServerInterceptor`, `grpc.StreamServerInterceptor`) cannot be used with Connect handlers. The type signatures are fundamentally different:

- **gRPC:** `func(ctx context.Context, req interface{}, info *grpc.UnaryServerInfo, handler grpc.UnaryHandler) (interface{}, error)` — works with raw `interface{}` and `grpc.UnaryServerInfo`.
- **Connect:** `func(next connect.UnaryFunc) connect.UnaryFunc` where `UnaryFunc = func(context.Context, connect.AnyRequest) (connect.AnyResponse, error)` — works with typed `AnyRequest`/`AnyResponse` wrappers providing headers, specs, and peer info.

Developers try to share interceptor logic between gRPC and Connect paths, discover they can't, then either duplicate code or build a broken abstraction layer.

**Why it happens:**
gaz v4.1 has a well-designed `InterceptorBundle` system (priority-ordered chains: logging=0, ratelimit=25, auth=50, validation=100, recovery=1000). The natural instinct is to reuse these bundles for Connect. But `InterceptorBundle` returns `[]grpc.UnaryServerInterceptor` — a type that Connect doesn't accept.

**How to avoid:**
1. Define a **protocol-agnostic middleware interface** at the `server/` level that captures cross-cutting concern *logic* (auth decisions, rate limit checks, logging context) without coupling to transport types.
2. Create **transport-specific adapters** that wrap the shared logic into `grpc.UnaryServerInterceptor` or `connect.Interceptor` respectively.
3. Keep the `InterceptorBundle` interface for gRPC, create a parallel `ConnectInterceptorBundle` for Connect, and have both delegate to shared middleware logic.
4. Do NOT try to write a single interceptor that works for both — the type systems are too different.

```go
// Shared logic (transport-agnostic)
type AuthChecker interface {
    CheckAuth(ctx context.Context, method string, metadata map[string][]string) (context.Context, error)
}

// gRPC adapter
func GRPCAuthInterceptor(checker AuthChecker) grpc.UnaryServerInterceptor { ... }

// Connect adapter  
func ConnectAuthInterceptor(checker AuthChecker) connect.UnaryInterceptorFunc { ... }
```

**Warning signs:**
- `InterceptorBundle` implementations referencing `google.golang.org/grpc/metadata` directly (transport coupling).
- Interceptors that extract gRPC-specific metadata (`md, ok := metadata.FromIncomingContext(ctx)`) — Connect uses `req.Header()` instead.
- Test suites that only test interceptors via gRPC calls, never via Connect calls.

**Phase to address:**
Phase 2 (Interceptor Unification) — must be done before Connect handlers are served, otherwise every service registration requires duplicating interceptor chains manually.

### Pitfall 2: Vanguard Transcoder Requires All Services Registered Upfront

**What goes wrong:**
`vanguard.NewTranscoder()` takes a `[]*vanguard.Service` slice at construction time. Once created, you cannot add more services. This conflicts with gaz's lazy DI resolution pattern where services are discovered via `di.ResolveAll[Registrar]` during `OnStart`.

If the transcoder is created before all `ConnectRegistrar` implementations are registered in DI, some services will silently not be transcoded — REST endpoints for those services will 404.

**Why it happens:**
gaz's DI container supports lazy resolution — providers run when first resolved, not at registration time. The gRPC server handles this fine because `grpc.NewServer()` accepts registrations incrementally via `RegisterService`. But Vanguard's `NewTranscoder()` is a one-shot construction.

**How to avoid:**
1. Make the Vanguard transcoder creation happen in an `OnStart` lifecycle hook, AFTER all DI providers have been triggered.
2. Use `di.ResolveAll[ConnectRegistrar]` first, collect all `(path, handler)` pairs, THEN construct the transcoder.
3. Mark the Vanguard server as **Eager** in DI (like the existing gRPC server) to ensure it starts in the correct order.
4. Add a startup validation: compare the set of gRPC-registered services with Vanguard-registered services, error on mismatch.

```go
// In OnStart, not in provider:
func (s *Server) OnStart(ctx context.Context) error {
    registrars, err := di.ResolveAll[ConnectRegistrar](s.container)
    // ... collect all handlers ...
    transcoder, err := vanguard.NewTranscoder(services)
    // ... mount transcoder ...
}
```

**Warning signs:**
- REST endpoints returning 404 while gRPC endpoints for the same service work fine.
- Transcoder creation in a DI provider function instead of a lifecycle hook.
- No validation comparing registered gRPC services vs. transcoded services.

**Phase to address:**
Phase 1 (Core Vanguard Integration) — this is foundational; getting the construction order wrong means nothing else works.

### Pitfall 3: h2c Configuration Breaks Existing gRPC Clients

**What goes wrong:**
Single-port serving (gRPC + Connect + REST on one port) requires HTTP/2 cleartext (h2c). Go 1.25+ supports `http.Protocols.SetUnencryptedHTTP2(true)`. But existing gRPC clients that connect with `grpc.WithTransportCredentials(insecure.NewCredentials())` may not negotiate h2c correctly with an `http.Server` (they expect a raw gRPC server, not HTTP/2 via `http.Server`).

The symptom is: gRPC clients that worked against the old `grpc.Server.Serve()` fail with connection errors against the new `http.Server` serving h2c.

**Why it happens:**
`grpc.Server` speaks native HTTP/2 (always). An `http.Server` serving h2c uses Go's HTTP/2 stack, which has subtly different behavior — particularly around connection preface handling, SETTINGS frames, and keepalive. Some gRPC client libraries (especially non-Go ones) are sensitive to these differences.

**How to avoid:**
1. Use `cmux` or protocol sniffing to route raw gRPC connections to `grpc.Server` and HTTP/1.1+h2c to the Vanguard `http.Handler`. The `vanguardgrpc` package exists specifically for wrapping an existing `grpc.Server`.
2. Alternatively, use `golang.org/x/net/http2/h2c` handler wrapper which is battle-tested: `h2c.NewHandler(mux, &http2.Server{})`.
3. Test with ALL client types: Go gRPC, grpcurl, grpc-web, curl, and any non-Go clients in use.
4. Add integration tests that verify gRPC reflection, health checks, and streaming all work through the h2c path.

**Warning signs:**
- gRPC clients connecting fine locally but failing in Docker/k8s (different HTTP/2 negotiation paths).
- Streaming RPCs hanging or dropping after upgrade.
- `grpcurl` working but programmatic gRPC clients failing (grpcurl uses a different HTTP/2 stack).
- Connection errors mentioning "protocol error", "GOAWAY", or "RST_STREAM".

**Phase to address:**
Phase 1 (Core Vanguard Integration) — must validate h2c compatibility before migrating any real traffic.

### Pitfall 4: Server Timeout Configuration Kills Streaming RPCs

**What goes wrong:**
`http.Server.ReadTimeout` and `WriteTimeout` apply to the **entire connection duration**, not individual requests. Setting `WriteTimeout: 30s` (common for REST APIs) will kill any streaming RPC that runs longer than 30 seconds. The existing gRPC server doesn't have this problem because `grpc.Server` manages timeouts per-RPC, not per-connection.

**Why it happens:**
When moving from `grpc.Server` to `http.Server` (required for Vanguard), developers carry over timeout configurations from the HTTP server config without realizing they now apply to gRPC streams too.

**How to avoid:**
1. Set `ReadTimeout` and `WriteTimeout` to 0 (disabled) on the `http.Server`.
2. Use per-request timeouts via context deadlines and gRPC/Connect timeout headers instead.
3. Implement an `http.Handler` middleware that applies timeouts selectively: short timeouts for REST/unary, no timeout for streaming.
4. Use `ReadHeaderTimeout` (which only applies to reading headers, not body) for DoS protection instead of `ReadTimeout`.

```go
server := &http.Server{
    ReadHeaderTimeout: 10 * time.Second, // DoS protection
    ReadTimeout:       0,                // Don't kill streams
    WriteTimeout:      0,                // Don't kill streams
    IdleTimeout:       120 * time.Second,
}
```

**Warning signs:**
- Streaming RPCs failing with "context deadline exceeded" after exactly N seconds.
- Unary RPCs working fine but server-streaming or bidi-streaming RPCs failing.
- Timeouts that only manifest under load (idle streams are fine, active streams break).

**Phase to address:**
Phase 1 (Core Vanguard Integration) — timeout config is part of the base `http.Server` setup.

### Pitfall 5: Breaking the Gateway Registrar Contract During Migration

**What goes wrong:**
gaz v4.1 has two `Registrar` interfaces:
- `server/grpc.Registrar` — registers gRPC services on `grpc.Server`
- `server/gateway.Registrar` — registers gateway handlers via `runtime.ServeMux`

Users implement BOTH for each service. Adding a third `ConnectRegistrar` means users must implement THREE interfaces during migration, or existing services silently lose REST transcoding.

If the migration isn't atomic (some services on old gateway, some on Vanguard), the system has TWO REST transcoding paths simultaneously — double the debugging surface.

**Why it happens:**
Incremental migration seems safer than big-bang. But partial migration creates a hybrid state where some REST calls go through grpc-gateway (with its codegen) and others through Vanguard (runtime transcoding) — with different error formats, different URL patterns, and different behavior.

**How to avoid:**
1. **Big-bang migration for REST transcoding:** Switch ALL services from grpc-gateway to Vanguard in one phase. Don't try incremental per-service migration.
2. **Keep both Registrar interfaces temporarily** but make `gateway.Registrar` optional with a deprecation notice. Services that only implement `grpc.Registrar` + `ConnectRegistrar` get transcoding via Vanguard automatically.
3. Use `vanguardgrpc.NewTranscoder(grpcServer)` initially — this wraps the existing gRPC server and gives Vanguard transcoding without ANY new registrar interface. Connect handlers come later.
4. Add compile-time interface checks: `var _ grpc.Registrar = (*MyService)(nil)` in service packages to catch missing implementations.

**Warning signs:**
- Some REST endpoints returning `{"error": "..."}` (grpc-gateway format) and others returning `{"code": "...", "message": "..."}` (Connect/Vanguard format).
- Gateway loopback connection still running alongside Vanguard.
- Services implementing only 1 of 2 required registrar interfaces.

**Phase to address:**
Phase 3 (Migration) — but planning starts in Phase 1 (deciding the migration strategy affects architecture).

### Pitfall 6: Losing gRPC-Gateway's Generated Code Benefits

**What goes wrong:**
grpc-gateway generates Go code from proto files + `google.api.http` annotations. This generated code handles:
- Field name mapping (proto snake_case → JSON camelCase)
- Query parameter parsing for GET requests
- Custom JSON marshaling via `runtime.JSONPb`
- Bidirectional field mask support

Vanguard does this at runtime instead. It mostly works, but edge cases differ — especially around:
- `google.protobuf.FieldMask` serialization (grpc-gateway has `FieldMaskMarshaler`)
- Custom `ServeHTTP` patterns in generated code
- `additional_bindings` in HTTP annotations
- Response body field selectors (`body: "result"` in annotations)

**Why it happens:**
Vanguard aims for compatibility with grpc-gateway's transcoding, but it's a different implementation. Subtle behavioral differences exist because grpc-gateway uses compile-time code generation while Vanguard uses runtime reflection on descriptors.

**How to avoid:**
1. Create a comprehensive REST API test suite BEFORE migration that tests every endpoint with exact request/response payloads.
2. Run the test suite against both grpc-gateway and Vanguard, diff the results.
3. Pay special attention to: `FieldMask` fields, `google.protobuf.Struct` fields, `oneof` fields in query params, repeated fields in query params, enum values as strings vs numbers.
4. Check Vanguard GitHub issues for known transcoding incompatibilities (#165: JSON codec options, #170: special characters in routes).

**Warning signs:**
- REST endpoint tests passing with grpc-gateway but failing with Vanguard.
- JSON field names changing (e.g., `user_name` appearing where `userName` was expected).
- Query parameters silently ignored instead of being parsed.
- `FieldMask` responses containing `user_name` instead of `userName` (or vice versa).

**Phase to address:**
Phase 3 (Migration) — requires running both systems in parallel during testing.

### Pitfall 7: Connect Health and Reflection Conflicting with Existing gRPC Services

**What goes wrong:**
gaz already has:
- gRPC health checking via `google.golang.org/grpc/health` (the `health/` package with `ShutdownCheck`)
- gRPC reflection via `google.golang.org/grpc/reflection`

Connect requires SEPARATE packages:
- `connectrpc.com/grpchealth` for health checking
- `connectrpc.com/grpcreflect` for reflection (both v1 and v1alpha)

Running both simultaneously on the same port means TWO health check endpoints and TWO reflection services — with potentially different state. The gRPC health server might report "SERVING" while the Connect health handler reports "NOT_SERVING" (or vice versa) during startup/shutdown races.

**Why it happens:**
Both gRPC-native and Connect health/reflection packages register on the same well-known paths (`grpc.health.v1.Health`, `grpc.reflection.v1.ServerReflection`). Vanguard routes requests to one or the other based on protocol detection, but state synchronization between them is the developer's responsibility.

**How to avoid:**
1. Use a **single source of truth** for health status. The existing `health.Manager` should drive BOTH the gRPC health server and the Connect health handler.
2. For reflection, use `connectrpc.com/grpcreflect` exclusively when serving via `http.Server` — it handles both gRPC and Connect clients. Remove the gRPC-native reflection if all traffic goes through Vanguard's `http.Handler`.
3. Wire the existing `ShutdownCheck` (which sets gRPC health to NOT_SERVING on shutdown signal) to also update Connect health status.
4. Test health checks with: `grpcurl`, `curl` (REST), and `connect-go` client — all three paths.

**Warning signs:**
- Health check endpoints returning different statuses depending on whether the client uses gRPC or HTTP.
- Load balancer health checks passing but gRPC clients seeing the service as unhealthy (or vice versa).
- Reflection listing different services depending on protocol.

**Phase to address:**
Phase 1 (Core Vanguard Integration) — health and reflection are infrastructure, must work from day one.

### Pitfall 8: Vanguard v0.x Instability in Production

**What goes wrong:**
Vanguard is currently at v0.4.0 — explicitly pre-stable. Known open issues include:
- #165: JSON codec options not fully respected in all code paths
- #184: Different codec used for error responses vs. success responses
- #170: Special characters in route paths break transcoding

Adopting a v0.x dependency in a framework means framework users inherit that instability. Bug reports come to gaz maintainers, not Vanguard maintainers.

**Why it happens:**
Vanguard is the only option for single-port gRPC + REST transcoding without code generation in Go. There's no stable alternative. The choice is: adopt v0.x with known issues, or stay on grpc-gateway.

**How to avoid:**
1. **Pin the exact Vanguard version** in `go.mod` — no `^` or floating versions.
2. **Wrap Vanguard behind a gaz interface** (`server/vanguard/transcoder.go`) so it can be swapped or patched without affecting users.
3. **Document known Vanguard limitations** in gaz's server package docs — users need to know what edge cases exist.
4. **Write regression tests** for each known Vanguard issue (#165, #170, #184) — when Vanguard releases a fix, tests should automatically pass.
5. **Have a fallback plan:** Keep the grpc-gateway module available (deprecated but functional) for users who hit Vanguard bugs.

**Warning signs:**
- Unexplained transcoding failures that aren't reproducible in unit tests.
- Error responses with unexpected JSON format (codec issue #184).
- Routes with special characters (e.g., URL-encoded path params) returning 404.
- Users reporting issues only with REST transcoding, never with direct gRPC.

**Phase to address:**
Phase 1 (Core Vanguard Integration) — version pinning and abstraction layer must be in the initial design.

## Technical Debt Patterns

| Shortcut | Immediate Benefit | Long-term Cost | When Acceptable |
|----------|-------------------|----------------|-----------------|
| Keep grpc-gateway alongside Vanguard permanently | No migration risk | Two REST transcoding systems, double maintenance, confusing for users | Never — deprecate gateway module by v5.1 |
| Duplicate interceptor logic for gRPC and Connect | Ships faster, no abstraction needed | Every interceptor change must be made twice, drift between gRPC and Connect behavior | Only for v5.0 MVP; unify by v5.1 |
| Skip `vanguardgrpc` wrapper, go straight to Connect handlers | Cleaner architecture | Requires all services to implement `ConnectRegistrar` immediately, big-bang migration | Only if ALL services can be migrated simultaneously |
| Use `grpc.Server` for gRPC + `http.Server` for Connect (two ports) | Avoids h2c complexity | Defeats the purpose of v5.0 (single-port), complicates deployment | Never — contradicts the milestone goal |
| Hardcode Vanguard service list instead of auto-discovery | Simpler initial implementation | Manual registration, breaks DI auto-discovery promise, services forgotten silently | Never — auto-discovery is a core gaz feature |

## Integration Gotchas

| Integration | Common Mistake | Correct Approach |
|-------------|----------------|------------------|
| Vanguard + CORS | Applying CORS middleware outside Vanguard — gRPC requests get CORS headers they shouldn't | Apply CORS only to the Vanguard handler (REST/Connect paths), not to the root mux. Use protocol detection or path-based routing. |
| Connect + gRPC metadata | Using `metadata.FromIncomingContext()` in Connect handlers — returns nil | Connect uses `req.Header()` for HTTP headers. Use `connect.Request.Header()` for Connect, `metadata.FromIncomingContext()` for gRPC. The interceptor adapter must translate between them. |
| Vanguard + OpenTelemetry | Using `otelgrpc` interceptors on the `http.Server` path — no spans generated | Use `otelhttp` middleware on the `http.Server`, `otelgrpc` on the `grpc.Server` (if still running), and `otelconnect` for Connect interceptors. Three different OTEL integrations for three protocols. |
| h2c + TLS termination | Enabling h2c on a server behind a TLS-terminating proxy — double encryption or protocol mismatch | When behind a proxy (nginx, envoy, cloud LB), the proxy handles TLS. The backend serves h2c. Ensure the proxy forwards HTTP/2 to the backend, not just HTTP/1.1. |
| Connect + streaming | Assuming Connect supports bidi streaming over HTTP/1.1 | Connect only supports bidi streaming over HTTP/2. Server-streaming works over HTTP/1.1 (via chunked transfer), but bidi requires h2c or TLS+HTTP/2. Document this limitation. |
| Vanguard + grpc-web | Assuming Vanguard handles grpc-web protocol | Vanguard transcodes REST↔gRPC. For grpc-web, use Connect's built-in grpc-web support or a separate grpc-web proxy. Don't conflate REST transcoding with grpc-web. |

## Performance Traps

| Trap | Symptoms | Prevention | When It Breaks |
|------|----------|------------|----------------|
| Vanguard re-serialization overhead | REST requests 2-3x slower than direct gRPC | Benchmark REST vs gRPC latency; Vanguard deserializes JSON → proto → re-serializes for gRPC. Accept this cost or use Connect clients directly for performance-critical paths. | Noticeable at >1000 req/s with large payloads |
| Single `http.Server` goroutine-per-connection | All traffic competing for the same goroutine pool; gRPC streams hold goroutines indefinitely | Set `http.Server.MaxConcurrentStreams` via HTTP/2 config. Monitor goroutine count. Use `ReadHeaderTimeout` + `IdleTimeout` to release idle connections. | >10k concurrent connections or >1k concurrent streams |
| Connect handler allocation per request | Connect creates new `connect.Request`/`connect.Response` wrappers per call; GC pressure at high throughput | Profile heap allocations. Consider sync.Pool for hot paths if Connect's allocation becomes a bottleneck. Usually fine up to ~50k req/s. | >50k req/s with small payloads (GC-bound) |
| Vanguard descriptor resolution on every request | Vanguard resolves proto descriptors and HTTP annotations per-request if not cached | Ensure proto files are compiled with `protoc-gen-go` and descriptors are embedded. Vanguard caches internally, but verify with profiling. | >5k req/s with many services (>50 methods) |

## Security Mistakes

| Mistake | Risk | Prevention |
|---------|------|------------|
| CORS allowing `*` origin on Vanguard handler | Any website can call your gRPC services via REST, bypassing browser same-origin protections | Configure CORS with explicit allowed origins. The existing `rs/cors` config in gateway must be migrated carefully to the new handler setup. |
| Auth interceptor only on gRPC path, not Connect | Connect/REST requests bypass authentication entirely | Auth must be applied at BOTH the gRPC interceptor level AND the Connect interceptor level (or as HTTP middleware above both). Test auth with all three protocols. |
| gRPC reflection exposed through Vanguard on public port | Attackers can enumerate all services, methods, and message types via REST calls to reflection endpoint | Disable reflection in production, or restrict it to internal networks. With single-port, reflection is accessible from the same port as REST APIs. |
| No rate limiting on REST transcoding path | REST APIs are easier to abuse than gRPC (curl vs grpc client); REST path may have no rate limiting if only gRPC interceptors enforce it | Apply rate limiting as HTTP middleware (above Vanguard) so it covers all protocols. Don't rely solely on gRPC interceptors. |

## DX Pitfalls

Developer experience issues specific to gaz framework users.

| Pitfall | Developer Impact | Better Approach |
|---------|-----------------|-----------------|
| Requiring users to implement 3 registrar interfaces (gRPC + Gateway + Connect) | Registration boilerplate triples; users forget one and get silent failures | Provide a single `ServiceRegistrar` interface that the framework adapts to all protocols. Or use `vanguardgrpc.NewTranscoder()` so users only implement `grpc.Registrar`. |
| Breaking `server.NewModule()` API without migration guide | All existing gaz users must rewrite server setup code | Provide `server.NewModule()` with the new behavior AND `server.NewLegacyModule()` for the old behavior. Deprecation warnings in v5.0, remove in v6.0. |
| Auto-discovery finding 0 Connect registrars and silently succeeding | Users think Connect is working but no handlers are registered | Log a WARNING when `di.ResolveAll[ConnectRegistrar]` returns empty. Or make it an error if Connect is explicitly enabled in config. |
| Different error formats between gRPC and REST responses | Frontend developers get inconsistent error shapes depending on which protocol path a request took | Standardize error format via a shared error handler. Connect's error model and gRPC's status codes map well, but JSON error shapes need explicit configuration. |

## "Looks Done But Isn't" Checklist

- [ ] **Single-port serving:** gRPC, Connect, and REST all work — but have you tested gRPC streaming over h2c? Unary works differently than streaming through `http.Server`.
- [ ] **Interceptor migration:** Auth interceptor works for Connect — but does it extract credentials from `req.Header()` instead of `metadata.FromIncomingContext()`? Test with actual Connect clients.
- [ ] **REST transcoding:** GET endpoints work — but do POST endpoints with `body: "*"` correctly deserialize JSON request bodies? Test with `FieldMask`, `Struct`, and `oneof` fields.
- [ ] **Health checks:** Health endpoint responds — but does it reflect shutdown state across BOTH gRPC and Connect health services? Kill the process and verify load balancers see NOT_SERVING before connections drain.
- [ ] **Reflection:** `grpcurl list` works — but does it list ALL services, including ones registered via Connect handlers? Compare against the gRPC-native reflection output.
- [ ] **CORS:** Browser requests succeed — but only for the configured origins? Test with an unauthorized origin and verify rejection. Ensure gRPC requests don't get CORS headers.
- [ ] **Graceful shutdown:** Server stops accepting new requests — but do in-flight streaming RPCs complete? The `http.Server.Shutdown()` behavior differs from `grpc.Server.GracefulStop()`.
- [ ] **Error mapping:** Connect errors have proper codes — but do REST-transcoded errors return appropriate HTTP status codes (400, 401, 403, 404, 500)? gRPC status codes map to HTTP codes differently than you might expect.
- [ ] **OpenTelemetry:** Traces appear — but do they have the correct service name, method name, and status code across all three protocols? Connect traces use different attribute names than gRPC traces.

## Recovery Strategies

| Pitfall | Recovery Cost | Recovery Steps |
|---------|---------------|----------------|
| Interceptor incompatibility discovered late | MEDIUM | Extract shared logic from existing interceptors into transport-agnostic functions. Write Connect adapters. ~2-3 days per interceptor type. |
| Vanguard service registration order wrong | LOW | Move transcoder creation to `OnStart` hook. Reorder DI resolution. ~1 day. |
| h2c breaks existing gRPC clients | HIGH | Fall back to `vanguardgrpc.NewTranscoder()` wrapping `grpc.Server`. Serve on two ports temporarily. Investigate h2c negotiation with affected clients. ~1 week. |
| Server timeouts killing streams | LOW | Set `ReadTimeout`/`WriteTimeout` to 0, add `ReadHeaderTimeout`. ~1 hour fix, but requires testing all stream types. |
| REST API behavior differs from grpc-gateway | MEDIUM | Write compatibility tests, file Vanguard issues, implement workarounds per-endpoint. ~3-5 days depending on number of affected endpoints. |
| Health check state inconsistency | LOW | Wire single health source to both gRPC and Connect health handlers. ~1 day. |
| Vanguard bug in production | HIGH | Roll back to grpc-gateway (must keep it as fallback). File issue upstream. Wait for fix or fork. ~1 day for rollback, weeks for upstream fix. |
| Auth bypass on Connect path | CRITICAL | Immediate hotfix: add HTTP-level auth middleware above all handlers. ~2 hours for hotfix, ~2 days for proper interceptor-level fix. |
