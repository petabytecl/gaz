    - Test: Publish after Close is a silent no-op (no panic, no delivery)
  </behavior>
  <action>
    In `eventbus/bus.go`, method `Close()` (line ~212):

    Current bug: Lock is released at line 225 (`b.mu.Unlock()`) BEFORE closing channels at lines 228-230. A concurrent `Publish()` holding RLock can see `closed=false`, collect handler refs, release RLock, then try to send on a channel that `Close()` just closed — panic.

    Fix: Keep `b.mu` write lock held while closing all channels. Only release the lock AFTER all `close(sub.ch)` calls complete. Then wait for handlers outside the lock:

    ```go
    func (b *EventBus) Close() {
        b.mu.Lock()
        if b.closed {
            b.mu.Unlock()
            return
        }
        b.closed = true

        var allSubs []*asyncSubscription
        for _, subs := range b.handlers {
            allSubs = append(allSubs, subs...)
        }

        // Close channels WHILE holding lock — prevents Publish from sending on closed channel
        for _, sub := range allSubs {
            close(sub.ch)
        }
        b.mu.Unlock()

        // Wait for handlers outside lock (they only read from ch, no lock needed)
        for _, sub := range allSubs {
            <-sub.done
        }

        b.logger.Info("eventbus stopped", "subscriptions_drained", len(allSubs))
    }
    ```

    Add test `TestEventBus_ConcurrentClosePublish` that spawns 50 goroutines publishing and 1 goroutine calling Close(), runs with -race flag, asserts no panic.
  </action>
