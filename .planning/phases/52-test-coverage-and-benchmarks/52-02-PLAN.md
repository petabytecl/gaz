    **Part B: Fix cron/internal test timing**

    The `cron/internal/cron_test.go` file uses `defaultWait = 1*time.Second + 50*time.Millisecond` and has many tests that `time.Sleep(2 * time.Second)` or more. The cron library's minimum schedule is 1 second (`@every 1s`), so we cannot reduce below 1s for schedule-based tests.

    Investigation strategy:
    1. Run `go test -v -count=1 ./cron/internal/ 2>&1` and capture per-test timing.
    2. Identify tests taking >2s each — these are the optimization targets.
    3. For tests that sleep waiting for cron to fire: check if the sleep can be replaced with a channel-based wait (cron job signals a channel when it runs, test blocks on channel with timeout instead of sleeping a fixed duration).
    4. For `TestStopAndWait` tests (~7s): these intentionally test stop behavior with slow jobs. Reduce the slow job duration from 3s to 500ms and adjust assertions.
    5. For `TestAddWhileRunningWithDelay` (~6s): reduce the schedule interval if possible.
    6. For `TestStartNoop`, `TestRunningJobTwice`, `TestNonLocalTimezone`, `TestLocalTimezone` (~2s each): replace `time.Sleep(defaultWait)` with channel-wait pattern where the job signals completion.

    Key constraint: This is a vendored/internal copy of robfig/cron. Changes must be careful and maintain test correctness. If a test genuinely needs real-time waiting (e.g., testing that cron fires at the right second), keep the sleep but document why.

    Target: total `cron/internal` test time under 15 seconds (from ~40s). If the investigation shows certain sleeps are unavoidable, document findings in a comment at top of file and accept partial improvement.

    Run `go test -race -count=1 ./cron/internal/` after changes to verify correctness.

  <action>
    Add `t.Parallel()` selectively to test functions that are truly independent. Follow these rules:

    **Safe to parallelize (DO add t.Parallel):**
    - Table-driven tests where each subtest creates its own fixtures (add to both parent and `t.Run` subtests)
    - Tests that create their own Container/EventBus/App instance with no shared state
    - Pure function tests (backoff calculations, config parsing)
    - Tests that use only local variables and no global state

    **NOT safe to parallelize (DO NOT add t.Parallel):**
    - Tests that set environment variables (`os.Setenv`)
    - Tests that write to shared files or directories
    - Tests that use a package-level variable as shared state
    - Tests that rely on timing (cron tests — skip these entirely)
    - Tests in testify suites (suite manages its own lifecycle)
    - Tests that modify global loggers or singletons

    **Process for each file:**
    1. Read the file completely
    2. For each top-level `func Test...` function, determine if it's independent
    3. If independent: add `t.Parallel()` as the first line after `func Test...(t *testing.T) {`
    4. For table-driven tests with `t.Run`: add `t.Parallel()` in both the parent test and each subtest. In subtests, capture the loop variable: `tc := tc` before `t.Run`.
    5. Run `go test -race ./package/` after each file to verify no races introduced

    **Target packages (prioritized by test count):**

    - `eventbus/` — bus_test.go (each test creates own EventBus)
    - `backoff/` — exponential_test.go (pure calculations)

    - Any test file where you're unsure about shared state
